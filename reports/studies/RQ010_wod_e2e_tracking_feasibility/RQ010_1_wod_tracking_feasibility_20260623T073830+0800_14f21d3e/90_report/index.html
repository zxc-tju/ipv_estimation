1. Research Context

The project's verifier estimates Interaction Preference Value (IPV) from motion and checks whether the implied behavior lies inside a human-normative envelope. The current downstream target is the counterpart-conditioned M3 verifier, not a context-only M2 substitution.

M3 requires the current and recent motion state of the relevant surrounding counterpart actor. Without counterpart tracks, IDs, histories, timing, and coordinate alignment, the verifier cannot estimate the counterpart-conditioned state needed for RQ010B. RQ010B is the later, rating-authorized candidate-preference validation task; this RQ010A report only decides whether the required data and infrastructure path exist.

2. Official Sources And Evidence Discipline

The source spine is the local official-source inventory and source findings: official_source_inventory.csv and source_findings.md. Load-bearing field, access, licence, timing, calibration, crosswalk, and scale claims are tied to official documents, release schemas, or official repository code. Primary-paper prose is retained only for context and benchmark-proxy discussion, not as the sole basis for release-field availability.

This discipline matters because the main negative finding is schema-level: the public E2ED schema documents camera/calibration, timestamp, ego state, and candidate-future fields, while it does not expose surrounding-actor trajectories or map geometry.

3. Access And Licence

Gate 010-0 is PASS. The audited Waymo terms create a licence-compatible path for non-commercial research, scientific publication, teaching, personal experimentation, and publication benchmarking, subject to attribution, non-redistribution, privacy, production-use, and institutional-authority restrictions.

The access path is still operationally gated: a human operator must sign in, agree to the licence when authorized, inspect the signed-in manifest, and record object sizes, shard counts, checksums, and download method before any payload download. Official compressed size, exact bucket/object paths, and minimum pilot pull granularity remain unavailable from the public unauthenticated pages.

Supporting files: data_access_and_license_audit.md and acquisition_checklist.md.

4. Schema And Field Availability

The requirements table and crosswalk show the asymmetry. Ego position, velocity, sampling, units, coordinate frame, history, candidate x/y futures, candidate horizon, candidate-to-score identity, and missing-label metadata are available or derivable from release-schema evidence. Counterpart actor ID, x/y, velocity, heading, history, validity, class, extent, visibility, route/reference, target path, selection identity, and current IPV require tracking. Map conflict geometry and route/reference lines are missing in the public E2ED schema.

The critical-frame current timestamp is available, but the original-run critical-frame index is requires_access_verification. Rater-level metadata is missing from the public schema. The feasibility result is therefore relative to rq009_interface_status=provisional_contract.

5. Tracking-Need Decision

The selected tier is T2_FULL_TRACKING_REQUIRED. T0_NO_TRACKING_NEEDED is rejected because public WOD-E2E release evidence does not expose surrounding-actor tracks, IDs, histories, classes, extents, validity, visibility, or uncertainty. T1_LIGHT_AUGMENTATION is rejected because no official E2E-to-WOMD/WOD crosswalk, official actor-track source, or E2E-aligned official perception output was verified.

T3_BLOCKED is not selected at this schema-decision stage because access/licence, timestamp, camera images, calibration, ego state, and released candidate-to-score identity are sufficient to define a T2 recovery path. It remains a real downstream risk if sequence grouping, calibration, critical-frame anchoring, map/reference requirements, or quality-gate criteria fail.

Route 4, adapting an existing multi-camera 3D/BEV tracker, is preferred because it uses the released camera/calibration/timestamp fields directly and is more reproducible than building the whole perception stack first. Route 5, a custom detection-association-BEV-tracking-map-uncertainty pipeline, is reserved as the fallback if Route 4 cannot ingest WOD-E2E, emit the M3 state contract, satisfy licence/provenance requirements, or pass the frozen blind gate.

6. Tracking Quality Gate

The tracking gate is outcome-blind. It sets ratings_read_allowed=false, forbids rating, IPV, deviation, and RQ010B outcome fields from sampling, thresholds, tracker tuning, QA, and adjudication, and uses the canonical pass/fail QA seed 2026062306. The previous P8 seed conflict was fixed before any manifest draw, payload access, tracker run, QA scoring, rating read, or outcome inspection.

Phase 11b added a concrete independent reference-annotation protocol: RQ010A_P11b_rating_blind_3d_bev_reference_v1. It requires a local/offline calibrated multi-view 3D/BEV annotation process, per-frame actor BEV centers, extents, class, visibility, persistent IDs, identity-switch timelines, continuity timelines, critical-frame counterpart consensus, reference covariance, and derived velocity from pre-critical BEV positions and verified timestamps. If the independent reference is absent, circular, rating-exposed, too uncertain, or reliability-failed, the affected hard metric is BLOCKED, never PASS.

Hard gates cover actor recall, BEV position error, velocity error, ID switches, continuity, critical-frame actor coverage, available pre-critical history, occlusion, conditional map/lane association, counterpart-selection agreement, and uncertainty calibration. MOTA, IDF1, and HOTA are report-only comparability metrics and cannot override a hard M3 failure.

7. Compute, Storage, And HPC

Phase 7 applies the FIX B3 decision matrix and returns BLOCKED_PENDING_ACCESS. The public evidence supports only derived logical planning ranges: full WOD-E2E compressed input about 0.15-17 TB, Route 4 decoded-image equivalent about 0.45-340 TB if materialized, intermediate storage about 0.5-120 TB, and full-run GPU demand about 20-4500 GPU-hours. These are not official manifest sizes and are not WOD-E2E measured throughput.

The conditional infrastructure expectation is that a small pilot may fit a single workstation GPU if shard access is modest and the model fits memory. A full run could become HPC_RECOMMENDED or HPC_REQUIRED after signed-in manifest and pilot benchmarks, but that is not the current label.

8. Pilot Benchmark Plan

The pilot is plan-only and requires explicit user authorization before sign-in, licence acceptance, manifest inspection, payload download, parser execution, tracker execution, IPV estimation, rating read, or quality-gate scoring. The pass/fail benchmark minimum is N=80 scenes, with a target of N=120 only if signed-in manifest sizes make the extra pull modest.

Sampling is rating-blind and uses seed 2026062306 with deterministic sorting by a SHA-256 key over the scene/frame identifier. The eligible manifest may include only allowed non-preference strata such as split, non-preference tags, ego intent, image/timestamp/calibration presence, sequence length, and hash decile. Human preference scores, score-derived order, rater metadata, IPV, deviation, tracker error after QA, and RQ010B outcomes are forbidden.

The pilot must measure official object sizes, pull granularity, sequence grouping, camera order, timestamps, calibration, parse/decode throughput, Route 4 GPU memory/FPS, intermediate storage, retry/abstention overhead, local versus HPC wall time, and the Phase 6 quality gates.

9. Candidate-Future Actor Protocol

The primary downstream protocol is a shared open-loop opportunity structure. For each scene, the same selected counterpart, same pre-critical counterpart history, same forecast or opportunity structure, same uncertainty representation, and same map/reference boundary are applied to all released ego candidate futures. This isolates the candidate ego geometry comparison instead of mixing candidate-specific simulated reactions into the primary test.

Candidate-conditioned actor forecasts are allowed only as a sensitivity analysis after the primary route is implemented and blind gates pass. Any actor predictor must be rating-free, have fixed provenance, carry explicit uncertainty, and must not use candidate scores, score-derived order, IPV deviations, or human feedback. Open-loop predicted actor response is not realised harm and must not be described as an actual actor reaction or causal safety outcome.

The N4 alignment table anchors all inputs at E2EDFrame.frame.timestamp_micros, treats the first candidate waypoint as after the current frame, cuts actor history at or before the critical frame, and requires camera-to-ego-to-M3 coordinate validation. If map/reference geometry is required by frozen RQ009 and cannot be supplied or bounded, M3 remains blocked.

11. Limitations And Boundaries

The core feasibility conclusion is bounded, not a direct M3 validation result. RQ010A did not download WOD-E2E, run a tracker, compute IPV, read ratings, or test preference association.
  • Map geometry missing: public E2ED schema does not expose map conflict geometry, lane centerlines, or route/reference polylines. This is a serious M3 boundary because IPV estimation relies on lane/reference context where required.
  • Critical-frame original-run index unverified: the current timestamp exists, but the original 20-second run index or sequence mapping must be verified after authorized access.
  • Rater-level metadata missing: the public schema exposes per-candidate preference-score metadata, not rater IDs or per-rater lists.
  • RQ009 interface provisional: all feasibility statements are relative to rq009_interface_status=provisional_contract and must be rechecked against the frozen interface.
  • Size and throughput derived: official WOD-E2E sizes, shard granularity, and Route 4 throughput remain sign-in-gated or unmeasured; current ranges are planning bounds only.
  • No M2 downgrade: a context-only M2 fallback may be reported only as a separate bounded analysis, not as full counterpart-conditioned M3 validation.

13. Final Feasibility Statement For Phase 12b

WOD-E2E is licence-compatible for non-commercial research and provides enough public release evidence to plan an M3 feasibility path, but it does not directly provide the counterpart-track and map/reference inputs needed for the primary counterpart-conditioned verifier. The direct M3 verifier is therefore not runnable from the public WOD-E2E release schema alone. The honest feasibility state is T2 with Route 4 preferred, Route 5 fallback, a frozen rating-blind quality gate, and a blocked-pending-access compute label until authorized manifest and pilot evidence replace the current derived ranges.

Phase 13 should perform final review and registration, refresh stale run status/traceability metadata, and keep the same boundaries visible: critical-frame original-run index unverified, map geometry missing, rater-level metadata missing, and RQ009 interface still provisional.
